---

火曜の午後。新人君が大量に印刷したNVMeの仕様書を机中に広げ、頭を抱えている。そこへ先輩が資料を取りに来る。

「先輩、ちょっと聞いてもいいですか…」

『どうした、その大量の書類は。まるで論文でも書くみたいな量だな。』

「これ全部<ruby>NVMe<rp>《</rp><rt>エヌブイエムイー</rt><rp>》</rp></ruby>の仕様書なんですけど、バージョン2.0と2.1って何が変わったんすか？バージョン番号が2に上がったってことは大きな変更かと思ったんですけど、見た感じあまり変わってないように見えて…」

『それが全然違うんだ。NVMe 2.0の一番の変更点は、それまで一枚岩だった**仕様書を完全分割**したことだ。』

「分割？どういうことですか？」

『従来のNVMe 1.xは一つの仕様書に全部書いてあったんだ。管理コマンドもI/Oコマンドもトランスポートも全部ひっくるめてな。NVMe 2.0ではそれを役割ごとに分割した。Base Specificationが共通の管理機能やキューイング機構を定義して、I/Oコマンドセットは個別のドキュメントに分かれたんだ。』

「なるほど。それでこの山のような書類ができたわけですね…Base Specに加えて、I/Oコマンドセットの仕様書が何冊もあると。」

『その通り。NVM Command Set、ZNS Command Set、KV Command Setの3つのI/Oコマンドセット仕様が独立して存在する。それぞれが異なるアクセスパターンに対応してる。管理コマンドもAdmin Command Setとして独立した。』

「NVMは従来のブロックI/Oですよね。で、<ruby>ZNS<rp>《</rp><rt>ゼットエヌエス</rt><rp>》</rp></ruby>と<ruby>KV<rp>《</rp><rt>ケイブイ</rt><rp>》</rp></ruby>が新しい。でも、なんで分割する必要があったんですか？一枚岩のままでも良かったんじゃ…」

『良い質問だ。理由は大きく二つある。一つは仕様の策定速度だ。一枚岩だと一部分の変更が全体に影響するから、改訂に時間がかかる。分割すれば各コマンドセットを独立して進化させられる。たとえばKVコマンドセットだけ改訂したい時に、NVMやZNSに影響を与えない。』

「なるほど。もう一つは？」

『もう一つはデバイス実装のしやすさだ。ZNS対応デバイスはZNS Command Setだけを実装すればいい。Base Specは共通だから、一から全部実装する必要がない。結果的に多様なデバイスが生まれやすくなる。NVMe 1.x時代は「NVMe準拠」と言うには全部の機能を実装する必要があったんだ。』

「つまり、全く異なるアクセスパターンを持つストレージデバイスを、単一のNVMeトランスポートで統一的に扱えるようになったってことですね。」

『その通り。トランスポート層は共通だから、下の物理層がPCIeでもTCPでもRDMAでも、上のコマンドセットは同じインターフェースで使える。これがNVMe 2.0の設計思想だ。』

「ZNSって聞いたことあります。ゾーン単位でシーケンシャル書き込みするやつですよね？」

『その通り。ゾーンは大規模なブロック単位で、シーケンシャル書き込み必須で、ゾーン全体のリセットで一括解放する。従来のSSDが内部でやってた**ガベージコレクション（GC）をホスト側に制御委譲**する思想だ。』

「GCって、NANDの消去がブロック単位でしかできないことに起因するアレですよね。有効データをコピーしてからブロックを消す…あれをホストが直接制御できるようになるってことですか。」

『そういうことだ。従来のSSDはデバイス内部でGCをやってたから、ホストからはいつGCが走ってるか分からなかった。GCが始まるとテールレイテンシが跳ね上がる。ZNSではホストが明示的にゾーンリセットコマンドを発行してゾーン全体を消去する。有効データのコピーが不要になるから、書き込みのムダが激減する。』

「最大の利点は書き込み増幅（WA）の大幅低減ってやつですね。通常SSDが1.5〜3倍のWAなのに対し、ZNSは1.1以下に抑えられる。」

『よく調べてるな。その通りだ。ホストが1GB書くのに対して、通常のSSDは内部で1.5〜3GBの書き込みが発生する。これがZNSだと実質1.1GB以下になる。寿命も性能も改善される。特に書き込み頻度の高いワークロードでは効果が絶大だ。』

「でもホスト側の負担が増えるんじゃないですか？ゾーンの管理を自分でやらないといけないんですよね。」

『そう。ゾーンへの書き込みは先頭から順番にしかできない。ランダム書き込み不可、上書き不可。だからデータのライフサイクルに合わせたゾーン管理が必要になる。Linuxだと<ruby>zonefs<rp>《</rp><rt>ゾーンフス</rt><rp>》</rp></ruby>や<ruby>F2FS<rp>《</rp><rt>エフツーエフエス</rt><rp>》</rp></ruby>が対応してる。SPDKみたいなユーザ空間フレームワークでもサポートが進んでる。あとファイルシステムレベルでZNSを隠蔽する研究も盛んだ。』

「じゃあ<ruby>KV<rp>《</rp><rt>ケイブイ</rt><rp>》</rp></ruby>コマンドセットはどういう発想なんですか？」

『キーバリューストアをNVMeプロトコル上で直接実現する仕様だ。Store、Retrieve、Deleteの3基本コマンドに加えて、キー列挙やバッチ操作もある。キー長もバリュー長も柔軟に指定できる。』

「Key-Valueって、ソフトウェアでやるのが普通じゃないですか？RocksDBとかRedisとか。なんでNVMeのレベルでやる必要があるんですか？」

『いい質問だ。ソフトウェアのKV層を経由すると、どうしてもオーバーヘッドが乗る。ブロックデバイスにKey-Valueを保存する場合、まずファイルシステムやKVSライブラリが論理ブロックアドレスに変換して、それをNVMeコマンドにする。NVMeデバイス自体にKVコマンドを理解させれば、この変換が不要になる。想定されてるのはNoSQLデータベースやCDNキャッシュ、機械学習の特徴量ストアだ。』

「CDNキャッシュ…エッジサーバで大量のKey-Valueを扱うユースケースなら、確かにNVMe直結の価値がありそうですね。」

『そう。特に大規模なKVストアだと、CPUのオーバーヘッド削減効果は無視できない。Storeコマンド一つでキーとバリューをデバイスに直接送り込める。ファイルシステムのオーバーヘッドもブロックレイヤのオーバーヘッドもなくなる。CDNのエッジサーバで何万台単位で導入すれば、CPUリソースの節約は莫大だ。』

「じゃあ2.1はどうなんですか？2025年に出たんですよね。」

『そう。<ruby>NVMe 2.1<rp>《</rp><rt>エヌブイエムイーにいてんいち</rt><rp>》</rp></ruby>では<ruby>FDP<rp>《</rp><rt>エフディーピー</rt><rp>》</rp></ruby>と<ruby>PMR<rp>《</rp><rt>ピーエムアール</rt><rp>》</rp></ruby>が追加された。』

「FDPからお願いします。」

『<ruby>FDP<rp>《</rp><rt>エフディーピー</rt><rp>》</rp></ruby>はFlexible Data Placementの略だ。ホストがデータの配置場所をヒントとしてデバイスに指示できる仕組みで、複数ストリームを効率的に分離してGC負荷とテールレイテンシを改善する。』

「ヒントって具体的にどういうことですか？」

『例えばデータベースのトランザクションログと実データは更新頻度が全く違う。ホストが「これはログだよ」「これは実データだよ」ってヒントを出せば、デバイスは別々のNANDブロックに配置できる。ログだけでガベージコレクションが発生しても、実データのブロックはGCの影響を受けない。テールレイテンシが安定するんだ。』

「なるほど…それってZNSと似てませんか？どっちもホストがデータ配置を制御するって点では。」

『似ているが目的が違う。ZNSはゾーン単位の完全な制御をホストに委譲するのに対し、FDPはヒントを与えるだけだ。デバイスが最終的な配置を決める。FDPの方が導入障壁が低い。既存のSSDでもファームウェア更新で対応できる可能性がある。ZNSはデバイスとコントローラの設計から見直す必要がある。』

「じゃあPMRの方は？」

『<ruby>PMR<rp>《</rp><rt>ピーエムアール</rt><rp>》</rp></ruby>はPersistent Memory Regionだ。NVMeデバイス上の不揮発性メモリ領域をPCIe経由で直接マップできる機能だ。不揮発性DIMMと似た使い方ができるが、通常のNVMe SSDと同じスロットで運用できる。』

「NVDIMMみたいなもんですか。でもNVDIMMって専用のメモリチャネルが必要ですよね？あれって結構特殊な構成で。」

『そう。PMRの利点は、通常のNVMe SSDと同じPCIeスロットで運用できることだ。専用のメモリチャネルが不要で、PCIeスロットに挿すだけで不揮発性メモリ領域が使える。高速な永続キャッシュやログ領域に使えるし、データベースのWrite Ahead LogをPMRに置く構成も研究されてる。』

「既存のNVMe SSDはどうなんですか？ファームウェアアップデートでZNS対応になったりするんですか？」

『それは無理だ。NANDの物理的な制約もあるし、コントローラの設計から変えないといけない。ただ、NVMe 2.xのドライバを実装したホストでも、NVMe 1.xのデバイスは今まで通りNVMコマンドセットで動作する。後方互換性はある。』

「どうやって使うコマンドセットを決めるんですか？」

『接続時に<ruby>Identify<rp>《</rp><rt>アイデンティファイ</rt><rp>》</rp></ruby>情報をやり取りして、対応しているコマンドセットをネゴシエーションする仕組みだ。デバイスが「俺はNVMとZNSに対応してる」と宣言して、ホストが適切なコマンドセットを選ぶ。ホストとデバイスの両方が対応してるコマンドセットだけが使える。接続時に自動で決まるから、ユーザが意識する必要はない。』

「なるほど…つまりNVMe 2.0/2.1って、新しい機能を追加したっていう以上に、NVMeの可能性を拡張する基盤を作ったって感じなんですね。」

『お、いいところに気づいたな。その通りだ。NVMe 1.xは「高速なSSDのためのプロトコル」だった。でもNVMe 2.0以降は、「多様なストレージデバイスのための統一プラットフォーム」に進化したんだ。』

「そういう視点で見ると、ZNSもKVもFDPもPMRも全部つながって見えてきますね。」

『そう。一枚岩だった仕様を分割したことで、それぞれのユースケースに特化したデバイスが生まれやすくなった。ZNS SSD、KV SSD、FDP対応SSD、PMR搭載SSD…それぞれが得意なワークロードに向かって最適化されていく。』

「つまり、自分のワークロードに合わせてコマンドセットを選ぶ時代になったってことか…それ、めちゃくちゃ面白いですね。」

『そう思うなら、まずは自分の担当システムのアクセスパターンを分析してみろ。ランダム書き込みが多いか、シーケンシャルか、Key-Value主体か。それによって最適なNVMeデバイスが変わる。』

「わかりました！早速、本番環境のI/Oパターンを解析してみます！」

『その意気だ。ところで、その仕様書の山、ちゃんと整理して片付けろよ。』

「あっ…はい、すみません！」

（こうして新人君は、NVMeが一枚岩の高速ストレージプロトコルから、多様なワークロードに最適化されたデバイス群を統一的に扱うプラットフォームへと進化していることを理解した。後に彼は、この知識を活かして自社のストレージ調達基準をワークロードベースで最適化するプロジェクトを立ち上げ、TCO削減と性能改善の両立を達成するのだった。）
